This guide demonstrates the basic steps required to create a reproducible project using RStudio and Quarto. Reproducible reporting ensures that analyses can be rerun and produce consistent results by combining code, output, and documentation in a single workflow.

The purpose of this guide is to establish the foundation for version control and collaborative practices using Git and GitHub.

1. Create a New RStudio Project and Render a Quarto File

Step 1: Create a New RStudio Project

A new RStudio Project was created to organise all files related to the analysis in a single folder. Creating a project ensures that all files are stored in one location and can be easily managed. It also supports reproducibility by keeping the working environment consistent.

To create a new project in RStudio:

  1. Open RStudio.
  2. Click File → New Project.
  3. Select New Directory.
  4. Select New Project.
  5. Enter a project name (for example, gitguide_project).
  6. Choose a location to save the project.
  7. Click Create Project.

After completing these steps, a new folder containing an .Rproj file will be created.

Step 2: Create a Quarto File Named example.qmd

Inside the project folder, create a new Quarto document named example.qmd, as following steps. The Quarto file allows code, text, and results to be combined into one document. This makes the analysis transparent and reproducible.

  1. To create the file, click File → New File → Quarto Document.
  2. Save the file as example.qmd in the project folder.
  3. Add YMAL at the top of the file between — (three dashes) to the file. YMAL controls the title, output format, table of content, theme, global settings of the file.
---
title: "Example Quarto Document" 
subtitle: "Part of ETC5513 Assignment2"
author: "Your Name" 
date: today
format: 
    html:
        theme: cosmo
        toc: true
    typst: 
        documentclass: article
        toc: true
---
  1. Text is written in markdown. Below are common markdown syntax.
  • Adding section and sub-section: Every # create a child sub-section of the parent section.

Markdown syntax:

# This is a section

Content goes here.

## This is a sub-section

More content here.
  • Adding text formatting: Below table is standard Markdown syntax.
Markdown syntax Result
**bold** bold
*italic* italic
`code` code
[link](https://example.com) link
~~strikethrough~~ strikethrough
  • Adding bullet points: Use - or *.

Markdown syntax: - unordered list 1 or * unordered list1

Result:

  • unordered list 1

  • Adding Numbered lists: Use 1., 2., etc. to represent first step, second step, etc.

Markdown syntax: 1. ordered list 1

  • Adding images: Use standard markdown image syntax:
![Caption text](path/to/image.png)

Or control the size with curly braces:

![Caption text](path/to/image.png){width=50%}

You can also use fig-align: center in the YAML or as a chunk option.

  1. Code chunks are fenced by delimeters, . Chunk output can be customised with chunk execution options, which are at the top of chunk, starting with #|
Option What it does

Step 3: Render the Quarto File into an HTML Document

To generate the HTML output, click the Render button in RStudio. After rendering, an HTML file named example.html will be created in the project folder. Rendering the document converts the source file into a final report that includes both the code and the output. This ensures that results are automatically updated whenever the code changes.

When the file is rendered successfully, the output will display the result of the Quarto file. This confirms that the Quarto document has been successfully knitted into an HTML file and demonstrates the integration of code and output within a reproducible report.

Your Folder Should Look Like

gitguide_project/
    example.qmd
    example.html
    gitguide_project.Rproj

2. Connect the Project to GitHub Using the Command Line Interface

The project folder (gitguide_project) was connected to a GitHub repository using the command line interface. This allows project files to be tracked, shared, and synchronised using version control.

Step 1: Create a Repository on GitHub

A new repository was created on GitHub to store the project files online. Creating a repository on GitHub provides a remote location to store and share project files. It allows version control to be used for collaboration and backup.

To create the repository:

  1. Open a web browser and go to https://github.com
  2. Click New repository
  3. Enter the repository name: gitguide_project
  4. Select Public
  5. Add a README file
  6. Click Create repository

Step 2: Open the Command Line Interface in RStudio

The terminal was opened in RStudio to run Git commands. The command line interface allows Git commands to be executed directly within the project folder, enabling version control operations.

To open the terminal: Tools → Terminal → New Terminal

Step 3: Clone the GitHub Repository

Cloning the repository creates a local copy of the GitHub repository and automatically sets up version control for the project.

The GitHub repository was downloaded to the local computer by the SSH key from GitHub using the following command: git clone https://github.com/username/gitguide_project.git

After running the command, a new folder named gitguide_project was created.

Step 4: Add Project Files to the Repository Folder

All project files must be inside the repository folder so that Git can track changes to those files. So, place files created in 1 into the cloned repository folder.

gitguide_project/
    example.qmd
    example.html
    gitguide_project.Rproj

Step 5: Add Files to the Repository

All files were added to the staging area using:

git add .

Adding files prepares them to be saved in the repository.

Step 6: Create a Commit

The files were updated to local repository using:

git commit -m "Add project files"

A commit records the current state of the project and creates a version history.

Step 7: Push the Files to GitHub

The files were uploaded to the remote GitHub repository (gitguide_project) using:

git push

Pushing synchronises the local project with the remote repository, ensuring that all files are stored online and can be shared.

Step 8: Check the Repository Status

The status of the repository was checked using:

git status

This command shows which files have been added or modified and confirms that Git is tracking the project.

The project folder was successfully connected to the GitHub repository. All project files were uploaded to GitHub, confirming that version control has been correctly established using the command line interface.

3. Create a New Branch and Update the Project

A new branch named testbranch was created to modify the example.qmd file without affecting the main version. Branching allows different versions of a project to be developed independently and supports collaboration in version control systems.

Step 1: Create a New Branch

A new branch called testbranch was created using the command line interface. Creating a branch allows changes to be made separately from the main project. This prevents accidental changes to the main version and allows different features to be developed independently.

git branch testbranch

Step 2: Switch to the New Branch

The working environment was switched to the new branch using below command. Switching branches ensures that any changes made will only affect the selected branch. This allows safe modification of project files.

git checkout testbranch

Step 3: Modify the File example.qmd

The file example.qmd was opened in RStudio and updated by adding new content. Below modification was added in the testbranch to demonstrate updating a file using version control.

### Additional Section
This is modification for testbranch branch

Step 4: Add the Updated File to the Repository

The modified file was added to the staging area using:

git add example.qmd

Adding the file prepares the updated version to be recorded in the project history.

Step 5: Create a Commit

Creating a commit records the changes and creates a new version of the project that can be tracked over time. The changes were saved as a new version of the project using:

git commit -m "Update example.qmd in testbranch"

Step 6: Push the Branch to the Remote Repository

The new branch (testbranch) and its changes were uploaded to the remote GitHub repository using:

git push origin testbranch

Pushing the branch ensures that the updated version of the project is stored on GitHub and can be accessed or reviewed by others.

Step 8: Check the Repository Status

The repository status was checked using below command. This command confirms that the file has been modified and shows which changes are ready to be saved.

git status

As a result, a new branch named testbranch was successfully created. The file example.qmd was modified, and the changes were saved in both the local repository and the remote GitHub repository. This demonstrates how version control can be used to manage updates safely using branches.
